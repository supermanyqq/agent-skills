# 按计划审查实现

## 核心定位

把“计划”当作审查契约，把“代码改动”当作被验收对象。目标不是帮忙补代码，而是判断实现是否忠实覆盖计划、是否引入缺陷或漏洞、验证是否充分，并给出可执行的阻塞项与观察项。

## 触发后先做

1. 收集计划来源：用户贴出的计划、任务清单、issue、PR 描述、设计文档、验收标准、TODO 清单或本轮对话中的实施方案。
2. 收集实现范围：优先读取 diff、变更文件、相关调用链、测试变更和验证输出；不要只看摘要。
3. 收集约束：读取适用的 `AGENTS.md`、仓库规范、语言/框架约定、安全红线和已有 review 规则。
4. 若计划缺失或不可定位，先要求用户提供计划；若只能从上下文推断，必须在结论里标为“不可完整验收”。
5. 除非用户明确要求修复，否则保持审查姿态，不直接改代码。

## 多视角审核

默认读取 `references/subagent-review-protocol.md`，并启动多个彼此独立的 subagent 做多视角审查。不要把 subagent 只作为复杂场景的可选增强；这是本 skill 的默认行为。

每个 subagent 只接收必要的计划、diff、相关文件路径和约束，不接收其他 subagent 的结论。若 subagent 工具未在当前工具列表中显式可用，先尝试通过工具发现能力查找 multi-agent/subagent 工具；只有确认当前环境无法启动 subagent 时，才按同样角色顺序做独立审查轮次，并在最终输出中说明这是“单 agent 模拟多视角”以及降级原因。

默认至少覆盖这些角度：

- 计划一致性：是否漏做、错做、超范围做、验收标准未满足。
- 缺陷与回归：逻辑、状态、边界条件、并发、错误处理、兼容性。
- 安全与隐私：鉴权、注入、路径/命令执行、密钥/PII 泄露、供应链与权限扩大。
- 架构与可维护性：分层、命名、契约、重复实现、文档注释、约束违规。
- 测试与验证：测试是否覆盖计划风险，命令是否真实执行，CI/本地验证缺口。

## 综合规则

- findings 必须以缺陷、风险和验收失败为中心，按严重度排序。
- 每条 findings 必须给出文件/行号或明确证据位置、对应计划项、影响、推理依据和修复方向。
- 区分“阻塞项”和“观察项”；不要把风格偏好包装成阻塞问题。
- 如果没有发现问题，要明确说“未发现阻塞项”，并列出仍然存在的验证盲区或残余风险。
- 不要泄露密钥、token、PII；如发现泄露，只描述位置和风险，不复述敏感值。

## 输出格式

使用中文输出，结构如下：

1. **结论**：`通过` / `有观察项通过` / `阻塞` / `无法完整评估`。
2. **阻塞项**：按 `P0/P1/P2/P3` 排序；无则写“未发现阻塞项”。
3. **观察项**：非阻塞但值得处理的问题；无则省略或写“无”。
4. **计划覆盖矩阵**：计划项、实现证据、状态（已覆盖/部分覆盖/未覆盖/超范围/无法验证）。
5. **漏洞与回归风险**：安全、隐私、数据损坏、兼容性、性能或并发风险。
6. **验证评估**：已看到的测试/命令、缺失验证、CI 风险。
7. **开放问题与假设**：影响验收结论的不确定点。

如果用户要求“像 argue 一样”，在结论中补充“多视角共识摘要”：哪些角色一致认为阻塞，哪些角色存在分歧，以及最终采纳理由。
