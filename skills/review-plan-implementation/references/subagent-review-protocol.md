# Subagent 审查协议

本协议是 `$review-plan-implementation` 的默认执行方式。启动 skill 后，默认开启多个彼此独立的 subagent 做多视角审查；只有当前环境确认无法启动 subagent 时，才允许降级为单 agent 按角色顺序模拟审查，并必须在最终结论中说明降级原因。目标是让多个审核视角先独立判断，再交叉质询，最后形成可追溯的验收结论。

## 输入隔离

给每个 subagent 的输入只包含：

- 计划或验收标准原文。
- 变更摘要、diff、相关文件路径和必要调用链。
- 适用约束，如 `AGENTS.md`、项目规范、语言/框架约定。
- 用户明确的审查目标。

不要把其他 subagent 的结论、主 agent 的怀疑点、预设答案或拟定修复方案提前给它。这样可以减少确认偏误。

## 推荐角色

### 1. 计划一致性审查员

关注问题：

- 每个计划项是否都有实现证据。
- 是否存在漏做、错做、语义偏移或验收标准未满足。
- 是否做了计划外改动，且这些改动是否改变行为、契约或风险面。
- 前后端、宿主/内核、配置/脚本等跨层要求是否同步处理。

输出重点：

- 计划项到代码证据的覆盖矩阵。
- 未覆盖、部分覆盖、超范围实现清单。

### 2. 缺陷与回归审查员

关注问题：

- 逻辑错误、状态机遗漏、边界条件、空值、异常路径。
- 并发、取消、重试、生命周期、资源释放。
- 兼容性、迁移、默认值、错误处理和用户可见行为。
- 改动是否破坏既有调用方或数据流。

输出重点：

- 可复现或可推导的 bug。
- 回归影响范围和最小修复方向。

### 3. 安全与隐私审查员

关注问题：

- 鉴权/授权绕过、权限扩大、路径穿越、命令注入、SQL/模板/脚本注入。
- 密钥、token、PII、内部路径或敏感上下文是否进入日志、错误、快照、示例或测试 fixture。
- 外部依赖、安装脚本、网络访问、反序列化和文件系统访问风险。
- 错误处理是否暴露敏感信息。

输出重点：

- 安全漏洞按可利用性和影响排序。
- 不复述敏感值，只说明位置、类型和风险。

### 4. 架构与可维护性审查员

关注问题：

- 分层边界、模块落点、命名、契约版本化是否符合项目约束。
- 是否出现重复实现、低语义 helper/util、临时绕路、协议字符串散落。
- 公共边界、状态流、错误语义、注释/Rustdoc 是否完整。
- 新依赖是否必要，许可证和安全影响是否说明。
- 是否存在魔法数字、魔法字符串、隐式约定或散落常量，导致语义难以追踪或后续修改容易漏改。
- 是否符合高内聚、低耦合、单一职责、关注点分离、封装边界清晰、依赖方向稳定等通用工程质量标准。
- 是否违反 DRY/KISS/YAGNI 等原则，例如无必要抽象、过度设计、重复分支、临时兼容逻辑长期化。
- 是否出现过长函数/文件、过深嵌套、过高圈复杂度、长参数列表、共享可变状态或难以测试的隐藏副作用。
- 是否保持可读性、可测试性、可扩展性和局部推理能力；修改一处是否会牵连过多无关模块。

输出重点：

- 违反架构约束的具体位置。
- 违反广泛认可工程质量标准的具体位置、影响范围和是否构成阻塞。
- 可维护性风险、后续演进成本和最小重构方向。

### 5. 测试与验证审查员

关注问题：

- 是否为新增或变更的生产逻辑补充了对应单元测试；默认必须有单元测试，缺失时按阻塞项处理，除非变更是纯文档、纯样式、纯配置且有明确理由。
- 单元测试是否贴近被测源码/模块约定，覆盖正常路径、失败路径、边界条件和关键状态转换，而不只是空跑或快照式存在。
- 单元测试是否能独立、稳定、可重复运行，不依赖外部网络、真实密钥、用户本机状态或测试顺序。
- 测试是否覆盖计划中的核心行为、失败路径、边界条件和回归风险。
- 本地验证命令是否与改动范围匹配，是否真实运行并通过。
- 是否缺少 CI 必检、文档生成、lint、格式化、安全扫描或覆盖率验证。
- 是否存在只验证 happy path 的假阳性。

输出重点：

- 明确列出新增/变更逻辑对应的单元测试文件、测试函数或测试用例；若没有，标为阻塞并说明必须补测的行为。
- 缺失测试/验证与对应风险。
- 阻塞交付的验证缺口。

## 交叉质询流程

1. **独立初审**：各角色先输出 findings、证据和严重度，不互相参考。
2. **质询轮**：把匿名化的 findings 摘要提供给其他角色，请它们标记：
   - `agree`：同意，证据充分。
   - `refine`：方向正确但严重度、范围或证据需要修正。
   - `dispute`：不同意，并说明反证。
3. **合并轮**：合并重复 findings；保留分歧，并标明最终采纳或降级理由。
4. **投票/共识**：
   - 任一 P0/P1 被两个及以上角色支持，默认判定为阻塞。
   - 单一角色提出的 P0/P1 若证据明确，也可判定为阻塞，但必须说明为什么不需要共识。
   - 只有缺少证据的猜测不得作为阻塞项。

## 严重度

- `P0`：会导致数据损坏、密钥/PII 泄露、权限绕过、远程执行、构建完全不可用或关键功能不可用。
- `P1`：明确违反计划核心目标、破坏公开契约、严重回归、关键验证缺失。
- `P2`：局部功能缺陷、边界行为错误、非关键验证缺口、可维护性风险较高。
- `P3`：轻微问题、命名/注释/测试增强建议、低风险观察项。

## Findings 模板

```text
[严重度] 标题
位置：/绝对/或/仓库/相对/路径:行号
对应计划项：计划编号或原文摘要
影响：用户可见影响、数据/安全/维护风险
证据：代码、diff、测试输出或调用链证据
修复方向：最小可行修复或需要补充的验证
共识：哪些角色支持、反对或要求降级
```

## 计划覆盖状态

- `已覆盖`：计划项有明确实现与验证证据。
- `部分覆盖`：实现了部分行为，但缺少边界、异常、跨层、文档或验证。
- `未覆盖`：未找到对应实现。
- `超范围`：实现了计划外行为，可能扩大风险或改变契约。
- `无法验证`：缺少计划、diff、文件访问或运行证据。
